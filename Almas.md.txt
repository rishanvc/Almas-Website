Chapter 1: Introduction
1.1 Purpose
The purpose of this Software Requirements Specification (SRS) is to define the requirements for the Almas Hospital Website. This document describes the features, functions, user roles, and overall behaviour of the system.
The website will serve as the official online platform for Almas Hospital, allowing visitors to access hospital information, departments, doctors, facilities, health packages, careers, contact details, and appointment services.
The system will also provide a dynamic content management system (CMS) that enables authorized users to manage website content without modifying the source code. All website content will be stored in the database and displayed dynamically.

1.2 Scope
The Almas Hospital Website is a dynamic web application designed to provide patients and visitors with easy access to hospital information and services through an interactive online platform.
The website allows visitors to browse hospital information, explore departments and their facilities, view doctor profiles, submit appointment requests, access career opportunities, view galleries, locate branches, and contact the hospital.
The system includes a secure administrative portal where authorized users can manage website content. Instead of updating pages manually, all website information is maintained through the content management system.
The website follows a content approval workflow. Content Creators can create or edit content, Content Approvers review the submitted changes, and approved content is automatically published on the website. The Administrator has complete control over all users, website content, approvals, and system settings.

1.3 Objectives
The main objectives of the Almas Hospital Website are:
* Provide an informative and user-friendly hospital website. 
* Present complete and up-to-date information about the hospital. 
* Allow visitors to explore hospital departments and department facilities. 
* Display doctor profiles and healthcare services. 
* Allow visitors to submit online appointment requests. 
* Provide career information and online job application facilities. 
* Display hospital branches, contact information, and location details. 
* Manage all website content dynamically through a content management system. 
* Implement a content approval workflow to ensure the accuracy of published information. 
* Reduce manual website maintenance by enabling content updates through the administrative portal. 
* Ensure secure access for different administrative users based on their roles. 





































Chapter 2: Overall Description
2.1 Product Perspective
The Almas Hospital Website Management System is a dynamic web-based application developed to provide an online platform for Almas Hospital. The system allows visitors to access hospital information, healthcare services, doctor profiles, department details, career opportunities, contact information, and other important resources.
Unlike a static website, all website content is managed dynamically through a Content Management System (CMS). Authorized users can update website content through a secure administration panel without modifying the website's source code. This allows the hospital to keep its information accurate, up-to-date, and easy to maintain.
The system also includes a content approval workflow to ensure that all published content is reviewed before it becomes visible on the public website.

2.2 Product Functions
The system provides the following major functions:
Public Website
* Display hospital information. 
* Display departments and department facilities. 
* Display doctor profiles. 
* Display health packages. 
* Display career opportunities. 
* Display gallery images. 
* Display branch information. 
* Allow visitors to submit appointment requests. 
* Allow visitors to send enquiries. 
* Allow visitors to search website content. 
* Display announcements and other dynamic website content. 
Administration Portal
* Manage users. 
* Manage website pages and content. 
* Manage departments. 
* Manage department facilities. 
* Manage doctor profiles. 
* Manage health packages. 
* Manage appointments. 
* Manage careers. 
* Manage gallery. 
* Manage branch information. 
* Manage contact enquiries. 
* Manage website settings. 
* Manage content approval requests. 

2.3 User Roles
The system consists of four user roles.
Administrator
The Administrator has full access to the system.
Responsibilities include:
* Manage users. 
* Manage website content. 
* Manage departments. 
* Manage doctors. 
* Manage appointments. 
* Manage careers. 
* Manage gallery. 
* Manage branches. 
* Manage website settings. 
* View and manage content approval requests. 
* Publish, update, or remove any website content. 

Content Creator
The Content Creator is responsible for creating and updating website content.
Responsibilities include:
* Create new website content. 
* Edit existing website content. 
* Update hospital information. 
* Update department information. 
* Update doctor profiles. 
* Update facilities. 
* Update career details. 
* Update gallery. 
* Submit content for approval. 
* View approval status. 
Content created or modified by the Content Creator is not published immediately. It is first submitted for approval.

Content Approver
The Content Approver reviews all submitted content before publication.
Responsibilities include:
* View pending approval requests. 
* Review submitted content. 
* Approve content. 
* Reject content with comments. 
* Publish approved content. 

Visitor
Visitors can access the public website without logging in.
Visitors can:
* Browse website information. 
* View departments and facilities. 
* View doctor profiles. 
* View health packages. 
* View careers. 
* View gallery. 
* Submit appointment requests. 
* Send enquiries. 
* Search website content. 

2.4 Operating Environment
The system is designed to run through a web browser and can be accessed using computers, laptops, tablets, and mobile devices with an internet connection.
The website should support modern web browsers, including:
* Google Chrome 
* Microsoft Edge 
* Mozilla Firefox 
* Apple Safari 
The administration portal is accessible only to authorized users after successful authentication.

2.6 System Workflow
The system follows a structured workflow for managing website content.
Content Management Workflow
1. The Content Creator creates or edits website content. 
2. The updated content is saved with a Pending Approval status. 
3. The Content Approver reviews the submitted content. 
4. If approved, the content is published on the website. 
5. If rejected, the content is returned to the Content Creator with comments for revision. 
6. The Administrator has full authority to manage users, content, approval requests, and website settings. 
This workflow ensures that all published content is reviewed before becoming visible on the public website.

Chapter 6: Non-Functional Requirements
6.1 Performance
The system should provide fast and efficient performance to ensure a smooth user experience.
The system shall:
* Load web pages within an acceptable response time under normal usage. 
* Support multiple users accessing the website simultaneously. 
* Process appointment requests and enquiries without noticeable delay. 
* Retrieve and display dynamic content efficiently.

6.2 Reliability
The system should provide reliable services with minimal interruptions.
The system shall:
* Operate continuously without unexpected failures. 
* Maintain consistent performance during normal operation. 
* Ensure that stored data remains accurate and reliable. 
* Recover from unexpected system failures with minimal data loss. 

6.3 Availability
The website should be available whenever users need to access it.
The system shall:
* Be accessible 24 hours a day, 7 days a week, except during scheduled maintenance. 
* Minimize system downtime. 
* Display appropriate maintenance messages when the website is temporarily unavailable.

The system shall:
* Support the addition of new departments, doctors, branches, and website pages. 
* Allow new features and modules to be added with minimal changes to the existing system. 
* Handle increasing amounts of website content and user activity efficiently. 

6.5 Usability
The system should be easy to use for both visitors and administrative users.
The system shall:
* Provide a simple and user-friendly interface. 
* Use clear navigation menus and page layouts. 
* Display meaningful success and error messages. 
* Ensure that administrative functions are easy to understand and operate. 
* Require minimal training for authorized users. 

6.6 Maintainability
The system should be easy to maintain and update.
The system shall:
* Allow website content to be updated through the Content Management System. 
* Support easy maintenance of departments, doctors, careers, galleries, and other website information. 
* Allow administrators to manage the website without modifying the application's source code. 
* Be designed to support future updates and enhancements. 

6.7 Accessibility
The website should be accessible to a wide range of users.
The system shall:
* Support access from desktop computers, laptops, tablets, and smartphones. 
* Display content clearly on different screen sizes. 
* Use readable fonts and appropriate color contrast.

6.8 Compatibility
The system should function correctly across commonly used platforms and browsers.
The system shall:
* Support modern web browsers, including Google Chrome, Mozilla Firefox, Microsoft Edge, and Apple Safari. 
* Display website content consistently across supported browsers. 
* Support different screen resolutions and device types. 

6.9 Data Integrity
The system should maintain the accuracy and consistency of stored information.
The system shall:
* Store all website content securely. 
* Prevent unauthorized modification of data. 
* Ensure that only approved content is published on the public website. 
* Maintain consistency between related data, such as departments, doctors, and facilities. 
































Chapter 7: Security Requirements
7.1 Overview
The Almas Hospital Website Management System shall implement appropriate security measures to protect website data, user accounts, and system resources. The system shall ensure that only authorized users can access administrative functions and that all website content is managed securely.

7.2 User Authentication
The system shall verify the identity of users before allowing access to the administration portal.
The system shall:
* Allow only registered users to log in. 
* Authenticate users using a valid username or email address and password. 
* Prevent unauthorized users from accessing the administration portal. 
* Allow users to log out securely. 

7.3 User Authorization
The system shall control user access based on assigned roles.
The system shall support the following roles:
* Administrator 
* Content Creator 
* Content Approver 
Each role shall have access only to the functions assigned to it.
Administrator
The Administrator shall have full access to all modules and system settings.
Content Creator
The Content Creator shall be allowed to:
* Create website content. 
* Edit website content. 
* Submit content for approval. 
* View the status of submitted requests. 
The Content Creator shall not publish content directly.
Content Approver
The Content Approver shall be allowed to:
* Review submitted content. 
* Approve content. 
* Reject content. 
* Publish approved content. 
The Content Approver shall not manage system users or website settings unless authorized by the Administrator.

7.4 Password Security
The system shall protect user passwords.
The system shall:
* Store passwords securely using encryption or hashing. 
* Prevent passwords from being displayed in plain text. 
* Allow users to change their passwords. 
* Require strong passwords for administrative users. 

7.5 Session Management
The system shall manage user sessions securely.
The system shall:
* Create a secure session after successful login. 
* End the session after logout. 
* Prevent unauthorized access to protected pages. 
* Redirect unauthenticated users to the login page. 

7.6 Content Approval Security
The content approval workflow shall ensure that only approved information is published on the website.
The system shall:
* Save newly created or edited content with a Pending Approval status. 
* Prevent pending content from appearing on the public website. 
* Allow only the Content Approver or Administrator to approve content. 
* Publish content only after approval. 
* Record the approval status of each content update. 

7.7 Input Validation
The system shall validate all user input before processing.
The system shall:
* Validate required fields. 
* Validate email addresses. 
* Validate phone numbers. 
* Restrict invalid characters where necessary. 
* Display appropriate validation messages for incorrect input. 

7.8 File Upload Security
The system shall secure uploaded files.
The system shall:
* Allow only supported file formats. 
* Restrict oversized files. 
* Store uploaded files securely. 
* Prevent unauthorized file access. 

7.9 Data Protection
The system shall protect all stored information.
The system shall:
* Prevent unauthorized modification of data. 
* Protect user information. 
* Protect appointment records. 
* Protect career applications. 
* Protect website content. 
* Protect uploaded documents and images.

Chapter 9: System Architecture
9.1 Overview
The Almas Hospital Website Management System is a web-based application that allows visitors to access hospital information and enables authorized users to manage website content through a secure administration portal.
The system follows a centralized architecture where all website information is stored in a database and displayed dynamically on the public website. Authorized users can create, edit, review, approve, and publish content through the Content Management System (CMS).

9.2 System Components
The system consists of the following main components:
Public Website
The public website is accessible to all visitors without requiring a login. It provides access to hospital information and online services.
It includes:
* Home 
* About Hospital 
* Chairman's Message 
* Mission & Vision 
* Departments 
* Department Facilities 
* Doctors 
* Health Packages 
* Gallery 
* Careers 
* Contact 
* Branches 
* Appointment Request 
* Search 

Administration Portal
The administration portal is accessible only to authorized users.
It provides modules for:
* User Management 
* Dynamic Website Content Management 
* Content Approval Management 
* Department Management 
* Doctor Management 
* Appointment Management 
* Career Management 
* Gallery Management 
* Contact Enquiry Management 
* Website Settings 

Database
The database stores all information required by the system.
This includes:
* User information 
* Website content 
* Departments 
* Department facilities 
* Doctors 
* Appointments 
* Careers 
* Job applications 
* Gallery 
* Contact enquiries 
* Branch information 
* Website settings 
* Content approval records 
All website pages retrieve their information dynamically from the database.

9.3 Dynamic Content Management
The website uses a dynamic Content Management System (CMS) to manage all website information.
Instead of modifying the application's source code, authorized users can update website content through the administration portal.
The CMS allows users to manage:
* Homepage content 
* Hospital information 
* Chairman's Message 
* Mission & Vision 
* Departments 
* Department Facilities 
* Doctors 
* Health Packages 
* Gallery 
* Careers 
* Branches 
* Contact information 
* Website settings 
Any approved changes are automatically reflected on the public website.

9.4 Content Approval Workflow
To maintain the accuracy and quality of published information, the system includes a content approval workflow.
The workflow is as follows:
1. A Content Creator creates or edits website content. 
2. The content is saved with a Pending Approval status. 
3. A Content Approver reviews the submitted content. 
4. If approved, the content is published on the website. 
5. If rejected, the content is returned to the Content Creator with comments for revision. 
6. The Administrator has full control over all content and approval activities. 
This workflow ensures that only reviewed and approved content is displayed to visitors.

9.5 Module Interaction
The modules within the system work together to provide a consistent user experience.
* Visitors interact with the public website to access hospital information and services. 
* Content Creators manage and update website content. 
* Content Approvers review and publish approved content. 
* Administrators manage users, content, system settings, and all operational activities. 
* The database stores and provides information to all modules as required. 


9.7 Content Publishing Process
The content publishing process ensures that only approved information appears on the public website.
The process is:
1. Content Creator creates or edits content. 
2. Content is submitted for approval. 
3. Content Approver reviews the content. 
4. Approved content is published. 
5. Rejected content is returned for modification. 
6. Published content becomes visible on the website immediately after approval.
